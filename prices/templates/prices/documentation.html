Technical Documentation

PerfumeX Prices knowledge base for developers, operators, and AI agents.

This page documents current mechanics from the codebase (Django app), not only UI behavior.

1) System Overview

  • Framework: Django (server-rendered templates + JSON endpoints for live search).
  • App process: Gunicorn via perfumex.service.
  • Web server: Nginx reverse proxy.
  • Database: PostgreSQL (production), SQLite may be used locally.
  • Email ingest: IMAP mailboxes + attachment parsing pipeline.
  • Scheduler: cron calling python manage.py import_emails through runner script.

2) Core Data Model

  • Supplier: supplier identity, matching patterns, defaults, per-supplier diagnostics.
  • Mailbox: IMAP credentials and host/port.
  • SupplierMailboxRule: mailbox-specific matching override (from/subject/filename).
  • ImportBatch: one email/manual import event for one supplier.
  • ImportFile: each imported attachment, parse status, hash, errors.
  • SupplierFileMapping: column mapping/profile per supplier + file kind.
  • SupplierProduct: current supplier catalog row (price, activity, linkage).
  • PriceSnapshot: historical point-in-time product price.
  • ExchangeRate: currency rates by date and pair.
  • ImportSettings: global import and scheduler settings.
  • EmailImportRun: run telemetry (counts, status, detailed log).
  • UserPreference: per-user UI settings (for example exclude terms).

Main relation path for traceability: PriceSnapshot -> ImportBatch -> ImportFile.

3) Import Pipelines

3.1 Manual Upload

  • Upload creates an ImportBatch + ImportFile.
  • File is parsed with active supplier mapping for selected file kind.
  • If parsing succeeds: file and batch become processed.
  • If parsing fails: file and batch become failed with error message.
  • Price snapshots are written during parse pipeline processing.

3.2 Email Import Command (manage.py import_emails)

  1. Recovers stale pending rows from interrupted old runs (auto-fail + optional retry from stored file).
  2. Checks import interval and enabled flag (unless --force).
  3. Syncs daily CBR USD/RUB rate if needed.
  4. Builds search window (since_date) from settings + last run rules.
  5. Runs run_import(...) across active mailboxes.
  6. Writes summary counters and updates ImportSettings.last_run_at.

3.3 Message and Attachment Matching

  • Search criteria can include FROM, SUBJECT, and date bounds.
  • For Gmail, importer can merge INBOX with All Mail folder for reliability.
  • Supplier matching priority: mailbox rule first, then supplier fallback patterns.
  • Attachment filename may be inferred from content-type when missing.
  • Blacklist terms skip files by filename (Import Settings).
  • Only supported price extensions are parsed: .xls, .xlsx, .csv.

3.4 Duplicate Detection

  • Content hash: SHA-256 of attachment payload.
  • Email import uses same-day dedupe mode: supplier + local day + hash.
  • Dedupe check is cross-mailbox for same supplier/day to prevent duplicates from multiple inboxes.
  • Both pending and processed are considered for same-day dedupe in one run.

4) Scheduler and Operations

  • UI path: Settings -> Import Settings.
  • Install scheduler writes runner script and cron entry with marker PERFUMEX_IMPORT_CRON.
  • Cron executes every 5 minutes, with flock lock and timeout guard.
  • Command itself respects configured interval (interval_minutes) and may skip if too recent.
  • Run now triggers a background forced run from the web app process.
  • Remove scheduler removes only marked cron line, keeps other cron entries untouched.

Runner script location: /opt/perfumex/run_import_emails.sh

Main runtime log: /var/log/perfumex_email_import.log

5) Search and Product List Mechanics

  • Supplier Products page defaults: status=active, currency=USD.
  • Query tokenizer matches by product name and supplier name tokens.
  • Per-user exclude terms are stored in UserPreference.supplier_exclude_terms.
  • Server list pagination is 100 items per page.
  • Live search endpoint returns paginated results and preserves current filters.
  • Currency conversion for list display uses latest available rates.

6) Product Detail, History, and Currencies

  • Detail page uses latest snapshot per local calendar day to build history series.
  • Chart supports views: original, USD, RUB.
  • Historical conversion uses rates for snapshot date (or nearest prior available date by service logic).
  • Table columns include original price and computed RUB/USD display values.
  • Each snapshot row links back to related import batch details for auditability.

7) Import Log and Traceability

  • Import Log can be filtered by supplier/status and sorted by date/updated/supplier/status/mailbox.
  • Detail page shows received timestamp, updated timestamp, supplier, mailbox, message ID, and file list.
  • Manual upload/backfill rows may have mailbox/message ID unavailable.
  • Message ID is the best key for cross-checking a specific email in mailbox diagnostics.

8) Failure Modes and Fast Debug Checklist

  1. Confirm app DB context: engine and DB name loaded from .env.
  2. Check migrations after deploy: schema mismatch causes no such column/table.
  3. Run force import once manually and inspect summary counters.
  4. Open import log details for failing batch and verify message ID and file status/error.
  5. Verify supplier match patterns: sender/subject/filename and mailbox rules.
  6. Verify cron installed and marker present in root crontab.
  7. Check timeout/lock behavior in runner logs.

tail -n 200 /var/log/perfumex_email_import.log
crontab -l

9) Development Rules for Safe Changes

  • Do not change import dedupe semantics without documenting migration impact on history.
  • Any search/list change must keep pagination and current filter state stable.
  • Any scheduler change must preserve lock + timeout to avoid overlapping runs.
  • Any currency conversion change must clearly separate latest-rate display vs historical-rate replay.
  • After UI changes, validate both desktop and mobile templates and run python manage.py check.
